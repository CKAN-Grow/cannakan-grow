# CSTP v1 Immutable Snapshot Migration Scope

## 1. Purpose

This document defines the exact minimal database migration scope for the first immutable CSTP snapshot schema implementation.

This is migration-scope planning only. It does not create migrations, implement schema, implement reports, implement certifications, expose CSTP publicly, modify operational APIs/UI, add automation, add breeder/source portals, integrate Community Grow, or integrate the Source Directory.

The goal is to establish the smallest safe internal database foundation for immutable CSTP report snapshots while preserving the existing operational CSTP workflow and the canonical Grow session system.

## 2. Scope Principles

The first immutable snapshot migration should be:

- additive only
- internal only
- isolated from public exposure
- compatible with existing CSTP operational tables
- compatible with future report generation
- compatible with future certifications without implementing them
- protective of `grow_sessions`
- designed for historical reproducibility
- designed for append-only audit lineage

The migration should create snapshot infrastructure, not a reporting product.

## 3. Minimal Safe Migration Slice

The minimal safe migration slice should include only what is required to persist immutable snapshot infrastructure.

Must exist in v1:

- immutable report root records
- immutable snapshot records
- frozen metric snapshot records
- immutable session snapshot references
- immutable audit linkage references
- internal status/version fields
- publication lock concepts
- supersession lineage references
- timestamps needed for reproducibility

Intentionally deferred:

- report rendering
- public report publication UI
- certification records
- public read APIs
- Source Directory exposure
- Community Grow filters
- automation
- breeder/source portals
- frontend trust systems

## 4. First-Slice Entity Concepts

The following entities are conceptual migration-scope targets. Exact SQL, field names, constraints, and indexes should be defined later in a draft migration.

### Immutable Report Root Records

Purpose:

- represent the internal parent report container for one CSTP test
- coordinate snapshot versions and future report lifecycle state
- provide a stable anchor for future publication and certification relationships

Minimum conceptual relationships:

- references `cstp_tests`
- may reference `sources`
- may reference the current prepared or published snapshot later

Minimum conceptual fields:

- report id
- CSTP test id
- source id
- current snapshot id
- report status
- internal visibility/state marker
- archived marker
- created at
- updated at

Boundary:

- report root records may remain mutable while internal
- report root records must not store published frozen metrics as mutable fields
- report root records do not create public reports in v1

### Immutable Snapshot Records

Purpose:

- preserve a frozen report snapshot version
- separate historical report evidence from mutable operational CSTP state
- provide the future source of truth for report publication

Minimum conceptual relationships:

- references immutable report root
- references `cstp_tests`
- may reference `sources`
- may reference prior/superseded snapshot ids
- may reference publication/audit events later

Minimum conceptual fields:

- snapshot id
- report id
- CSTP test id
- source id
- snapshot version
- snapshot status
- generated at
- prepared at
- published at
- immutable lock marker
- supersedes snapshot id
- superseded by snapshot id
- report schema version
- methodology version
- created at

Boundary:

- published snapshot records never mutate
- superseded snapshots remain accessible
- draft/prepared snapshots may be regenerated by creating a new version according to a later workflow
- no public read behavior is created in v1

### Frozen Metric Snapshot Records

Purpose:

- freeze reportable CSTP metrics for one immutable snapshot
- prevent published report values from being recalculated from live session data
- preserve enough context for reproducibility and future certification eligibility

Minimum conceptual relationships:

- references immutable snapshot
- references immutable report root
- references `cstp_tests`

Minimum conceptual fields:

- metric snapshot id
- snapshot id
- report id
- CSTP test id
- metric key or metric group
- metric value
- metric unit/type
- numerator
- denominator
- calculated at
- observation window start
- observation window end
- calculation/method version
- created at

Boundary:

- metric records tied to published snapshots never mutate
- metric corrections require new snapshot lineage later
- certification thresholds are not implemented in v1

### Immutable Session Snapshot References

Purpose:

- freeze which CSTP session links and Grow sessions were included in one snapshot
- preserve relationship context without duplicating Grow session ownership
- support reproducible report evidence

Minimum conceptual relationships:

- references immutable snapshot
- references immutable report root
- references `cstp_test_sessions`
- references `grow_sessions`

Minimum conceptual fields:

- session snapshot id
- snapshot id
- report id
- CSTP test id
- CSTP test session id
- grow session id
- KAN label
- included in report marker
- relationship archived state at snapshot time
- session snapshot metadata
- created at

Boundary:

- `grow_sessions` remain canonical
- session snapshot records do not mutate Grow sessions
- session snapshot records do not fork session lifecycle, timeline, analytics, notes, media, reminders, or partition behavior

### Immutable Audit Linkage References

Purpose:

- preserve the audit trail between report/snapshot actions and admin actor history
- support future publication accountability
- connect snapshot records to append-only CSTP admin events

Minimum conceptual relationships:

- references immutable report root
- references immutable snapshot
- may reference `cstp_admin_events`
- may reference admin actor/user identity through existing event records

Minimum conceptual fields:

- audit linkage id
- report id
- snapshot id
- CSTP admin event id
- event role
- created at

Boundary:

- audit linkage remains internal-only
- audit events remain append-oriented
- publication should not be considered complete later without audit linkage
- no public audit exposure is created in v1

## 5. Explicitly Not Included In First Migration

The first immutable snapshot migration must not include:

- certifications
- certification history
- public visibility rollout
- public report APIs
- public report UI
- report publication UI
- report renderer
- PDF/export generation
- media storage systems
- new file upload systems
- Community Grow integration
- Source Directory integration
- public badges
- public trust scoring
- automation
- notification systems
- breeder/source portals
- frontend trust systems
- RLS/public policy implementation
- changes to existing Grow session behavior
- changes to existing operational CSTP APIs/UI

These are future layers and should remain deferred.

## 6. Required Immutability Rules

The migration scope must support these rules conceptually:

- published snapshot records never mutate
- frozen metric records tied to published snapshots never mutate
- session snapshot references tied to published snapshots never mutate
- superseded snapshot lineage remains accessible
- operational CSTP entities remain mutable internally
- report root state must not overwrite published snapshot evidence
- corrections/amendments create new snapshot lineage later
- `grow_sessions` remain canonical operational entities
- CSTP does not create session forks
- audit lineage remains append-oriented

The first migration should include enough fields to support these rules even if locking enforcement is implemented in helper/service logic later.

## 7. Relationship Boundaries

### Operational CSTP to Immutable Snapshots

Operational CSTP tables remain the internal workflow system:

```text
cstp_requests
-> cstp_tests
-> cstp_test_sessions
-> grow_sessions
```

Immutable snapshots consume this state and freeze report evidence:

```text
cstp_tests
-> immutable report root
-> immutable snapshot
-> frozen metrics / session references / audit linkage
```

Operational records may continue to change internally. Published snapshot evidence must remain stable.

### Immutable Snapshots to Future Reports

Future report rendering should read from immutable snapshots, not live operational records.

Deferred future flow:

```text
immutable snapshot
-> report generation/rendering
-> publication workflow
-> public report API/UI later
```

This future layer is not part of the first migration.

### Immutable Snapshots to Future Certifications

Future certifications should depend on immutable snapshots.

Deferred future flow:

```text
immutable snapshot
-> certification eligibility
-> certification record
-> certification history
-> public badge/report availability later
```

Certification schema and certification logic are not part of the first migration.

## 8. Migration Risks to Avoid

### Coupling Operational and Immutable State

Avoid storing frozen report truth only on mutable operational CSTP records. Operational state and report evidence must remain separate.

### Mutable Snapshot Records

Avoid a design where published snapshot rows can be updated in place without version or audit lineage.

### Embedding Public Trust Logic Too Early

Avoid public visibility, certification levels, public badges, public trust scoring, or Source Directory/Community Grow exposure in the first schema slice.

### Duplicating Lifecycle Systems

Avoid creating a parallel CSTP request/test lifecycle inside report tables. Reports can have report/snapshot statuses, but they should not redefine operational request/test truth.

### Premature Certification Schema

Avoid adding certification tables before immutable snapshot behavior is validated.

### Grow Session Forking

Avoid copying full Grow session records into CSTP-owned session tables or treating report session references as the new session source of truth.

### Unsafe Deletes

Avoid cascading deletes that could remove historical report roots, snapshots, metric records, session references, or audit linkage needed for reproducibility.

## 9. Readiness Criteria Before Draft SQL

Before SQL is drafted, engineering should confirm:

- final conceptual table names
- final FK targets
- nullable vs required relationships
- delete behavior expectations
- snapshot versioning model
- publication lock model
- report/snapshot status vocabulary
- metric storage strategy
- session reference storage strategy
- audit linkage strategy
- expected indexes/query paths
- internal-only visibility model

## 10. Final Recommendation

The first immutable CSTP snapshot migration should be narrowly scoped to internal immutable snapshot infrastructure.

It should establish report roots, snapshot versions, frozen metric records, session snapshot references, and audit linkage references. It should not implement report generation, certifications, public APIs/UI, public trust systems, automation, breeder/source portals, Community Grow integration, or Source Directory integration.

Once this migration scope is reviewed, the next step should be an executable-quality draft SQL candidate in a drafts folder only, followed by local isolated validation before any active migration is created.
